## Usage

Import the plugin from your framework's package:

```typescript
// Vue — vite.config.ts
import fluentiVue from '@fluenti/vue/vite-plugin'
export default defineConfig({ plugins: [vue(), fluentiVue()] })

// React — vite.config.ts
import fluentiReact from '@fluenti/react/vite-plugin'
export default defineConfig({ plugins: [fluentiReact(), react()] })

// Solid — vite.config.ts
import fluentiSolid from '@fluenti/solid/vite-plugin'
export default defineConfig({ plugins: [solidPlugin(), fluentiSolid()] })
```

## Options

The plugin accepts a single `config` option — either a path to a config file or an inline `FluentiConfig` object. When omitted, the plugin auto-discovers `fluenti.config.ts` in the project root.

```typescript
// Zero-config — auto-discovers fluenti.config.ts
fluentiVue()

// Point to a specific config file
fluentiVue({ config: './i18n/fluenti.config.ts' })

// Inline config override
fluentiVue({
  config: {
    compileOutDir: './src/locales/compiled',
    splitting: 'dynamic',
  },
})
```

<Aside type="tip">
Most projects only need `fluentiVue()` with a `fluenti.config.ts` at the project root. Inline config is useful for monorepos or custom setups where you need per-app overrides.
</Aside>

### `config` fields

The following fields are accepted when passing an inline config object. All fields are optional — defaults come from `fluenti.config.ts` or the built-in defaults shown below.

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `sourceLocale` | `string` | `'en'` | Source locale used for descriptor fallback and build transforms |
| `locales` | `LocaleDefinition[]` | `['en']` | Locale codes or objects (`string \| { code, name?, iso?, dir?, domain? }`) available to the build |
| `format` | `'json' \| 'po'` | `'po'` | Catalog file format |
| `compileOutDir` | `string` | `'./src/locales/compiled'` | Directory where compiled catalog files are read from |
| `splitting` | `'dynamic' \| 'static' \| false` | `false` | Build-time locale chunking strategy |
| `defaultBuildLocale` | `string` | `sourceLocale` | Locale inlined for `'static'` builds |
| `include` | `string[]` | `['./src/**/*.{vue,tsx,jsx,ts,js}']` | Source file patterns for dev watcher |
| `exclude` | `string[]` | `['**/*.test.*', '**/*.spec.*', '**/__tests__/**', '**/*.d.ts']` | File patterns excluded from dev watcher |
| `devAutoCompile` | `boolean` | `true` | Auto extract+compile on file change during dev |
| `buildAutoCompile` | `boolean` | `true` | Auto extract+compile before production build |
| `devAutoCompileDelay` | `number` | `300` | Debounce delay (ms) for dev auto-compile |
| `parallelCompile` | `boolean` | `false` | Use worker threads for compilation (effective with 5+ locales) |
| `catalogExtension` | `string` | `'.js'` | File extension for compiled catalog files (e.g. `'.mjs'`, `'.json'`) |
| `idGenerator` | `(message: string, context?: string) => string` | built-in FNV-1a | Custom message ID generator. Replaces the default hash-based ID strategy |
| `onBeforeCompile` | `() => boolean \| void \| Promise<…>` | — | Hook called before auto-compile. Return `false` to skip compilation |
| `onAfterCompile` | `() => void \| Promise<void>` | — | Hook called after auto-compile completes |

## What it does

The plugin provides several sub-plugins, each implemented as a dedicated module:

The core `@fluenti/vite-plugin` provides these sub-plugins:

1. **fluenti:virtual** — Internal virtual-module layer used by the build output (`virtual-modules.ts`)
2. **fluenti:script-transform** — Strict AST-only tagged-template optimization (`scope-transform.ts`, `trans-transform.ts`)
3. **fluenti:build-compile** — Auto extract+compile before production builds
4. **fluenti:build-split** — Code-splitting transforms for production builds (`build-transform.ts`)
5. **fluenti:dev** — HMR support for catalog changes

Each framework package adds its own plugins:

- `@fluenti/vue/vite-plugin` adds **fluenti:vue-template** — pre-transforms Vue `v-t`, `<Trans>`, and `<Plural>` template usage
- `@fluenti/react/vite-plugin` and `@fluenti/solid/vite-plugin` pass the core plugins through with the correct framework-specific runtime generator

## Splitting strategies

Use `splitting: 'dynamic'` for locale chunks loaded on demand, `splitting: 'static'` for a single inlined build locale, or `false` to bundle every catalog together.

The plugin's internal virtual modules (`virtual:fluenti/runtime`, `virtual:fluenti/messages`, `virtual:fluenti/route-runtime`) are implementation details, not the primary public API. App code should use framework runtime APIs such as `setLocale()`, `preloadLocale()`, `loadMessages()`, and `chunkLoader`.

## Script transforms

The plugin uses **AST scope analysis** to detect Fluenti's direct-import `t` inside supported component/setup scopes and rewrites it at build time. Other variables named `t` are left untouched.

```ts
import { t } from '@fluenti/react'
const greeting = t`Hello ${name}`
```

```ts
const { t } = useI18n()
const greeting = t('Hello {name}', { name: name })
```

Supported compile-time forms:

- `` import { t } from '@fluenti/react'; t`Hello ${name}` ``
- `import { t } from '@fluenti/vue'`
- `import { t } from '@fluenti/solid'`
- `t({ message: 'Hello {name}', context: 'hero' }, { name })`

Runtime lookup such as `t('message.id')` still belongs to `useI18n().t()`.

## `<Trans>` compile-time optimization

The plugin also optimizes JSX `<Trans>` components at build time. It statically extracts the message string from children and pre-computes the context-aware message ID, injecting `__id`, `__message`, and `__components` props. This eliminates the runtime `extractMessage()` / `hashMessage()` fast-path cost without changing the component API.
